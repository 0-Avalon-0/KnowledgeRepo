# Git 规范

这份规范适用于当前知识库仓库，目标不是追求复杂流程，而是让“原始资料导入、知识库整理、派生产出、检查与工具”在版本历史里保持清晰可追踪。

## 目标

这套 Git 规则主要解决四个问题：

1. 让每次变更都能看出是在导入资料、整理知识，还是生成结果。
2. 避免把大量不相关修改混进同一个提交。
3. 让后续回看历史时，能快速定位某个主题是何时加入、何时整理、何时被修订。
4. 让代理和人工协作时，双方都能按同一套结构提交。

## 分支规则

默认分支使用 `main`，表示当前仓库中相对稳定、可继续积累的状态。

推荐只保留少量短生命周期分支，避免把知识库仓库搞成复杂的软件开发流。

推荐分支命名：

1. `main`
用途：稳定主线，保留已经确认可保留的资料、知识条目和派生产物。

2. `ingest/<日期>-<主题>`
用途：导入一批新的原始资料。
示例：`ingest/2026-04-09-rag-papers`

3. `compile/<主题>`
用途：把原始资料整理为来源笔记、概念页、主题页或摘要。
示例：`compile/retrieval-augmented-generation`

4. `derive/<主题>`
用途：围绕某个问题生成回答、报告、幻灯片或可视化。
示例：`derive/long-context-vs-rag`

5. `check/<日期>`
用途：做健康检查、去重、修复断链、清理索引。
示例：`check/2026-04-09`

6. `tool/<主题>`
用途：修改脚本、命令行工具、自动化流程或 skill。
示例：`tool/knowledge-repo-skill`

## 什么时候需要新分支

下面这些情况建议新开分支：

1. 一次性导入很多资料，可能会连带产生大量来源笔记和索引更新。
2. 准备对某个主题做较大规模整理。
3. 需要试验新的目录结构、脚本或工作流。
4. 要生成一批阶段性报告，但还不确定是否保留。

下面这些情况可以直接在 `main` 上进行：

1. 修正一个错别字。
2. 补一条小的来源链接。
3. 更新一个很小的摘要。
4. 增加一个很小的检查结果文件。

如果你拿不准，一个简单原则就够了：

1. 修改少且风险低，直接在 `main`。
2. 修改多、范围广、还需要筛选，开分支。

## 提交类型

这个仓库不必照搬软件工程里很重的 Conventional Commits，但最好保留清晰前缀。推荐使用下面这些前缀：

1. `ingest:`
用于导入原始资料。
示例：`ingest: 导入 4 篇关于 RAG 的论文与网页文章`

2. `compile:`
用于把原始资料整理为来源笔记、概念页、主题页、摘要。
示例：`compile: 新增检索增强生成主题页和相关概念条目`

3. `derive:`
用于生成回答、报告、幻灯片、图表等派生内容。
示例：`derive: 生成长上下文与检索系统对比回答`

4. `check:`
用于健康检查、去重、断链修复、索引修复。
示例：`check: 补充孤立页面检查结果并修复索引链接`

5. `tool:`
用于工具、脚本、skill、仓库自动化相关修改。
示例：`tool: 新增知识库工作流 skill`

6. `docs:`
用于修改仓库说明、流程文档、规范文档。
示例：`docs: 补充 Git 使用规范与分支规则`

7. `chore:`
用于不属于以上类别的小型整理工作。
示例：`chore: 调整目录占位文件和文本属性`

## 提交粒度规则

尽量遵守下面的拆分原则：

1. 原始资料导入和知识库编译，尽量分两个提交。
2. 概念页更新和主题页重写，规模较大时尽量分开提交。
3. 派生内容如果只是一次性输出，不要和长期知识条目混在同一个提交里。
4. 工具或 skill 的修改，不要和知识内容修改混在一起。
5. 健康检查结果和真正的内容修复，最好拆开成两个提交。

反例：

1. 同一个提交里同时包含 20 篇原始文章、5 个概念页重写、1 个脚本改动、3 份报告输出。

正例：

1. `ingest: 导入上下文工程相关文章与截图`
2. `compile: 为上下文工程新增来源笔记和概念条目`
3. `derive: 生成上下文工程学习总结`

## 推荐日常流程

如果你今天新增了一批资料，推荐按下面流程走：

1. 新建分支，例如 `ingest/2026-04-09-context-engineering`。
2. 提交原始资料导入。
3. 让代理生成或更新来源笔记。
4. 提交来源笔记和索引更新。
5. 如果进一步整理出概念和主题，再单独提交。
6. 如果生成了报告或回答，再单独提交派生内容。
7. 检查结果满意后，再合并回 `main`。

如果你今天只是修一点内容，推荐流程更简单：

1. 直接在 `main` 修改。
2. 用一个清晰提交说明完成保存。

## 适合这个仓库的提交模板

你可以直接照这个句式写提交信息：

```text
<类型>: <这次变更做了什么>
```

例子：

```text
ingest: 导入 6 篇关于模型蒸馏的原始资料
```

```text
compile: 新增模型蒸馏来源笔记并更新概念页
```

```text
derive: 生成模型蒸馏研究简报
```

```text
check: 修复孤立主题页并更新待整理清单
```

## 不建议提交的内容

1. 编辑器本地状态。
2. 临时缓存文件。
3. 未筛选的大量无意义中间产物。
4. 明显重复、尚未命名清楚的碎片文件。
5. 含有敏感信息、账号信息、私人数据的原始材料。

## 一个简单判断标准

如果你不确定这次变更该怎么提交，就问自己两个问题：

1. 这次改动主要是在导入资料、整理知识、生成结果，还是改工具？
2. 一个月后回看历史时，我是否能从提交信息里立刻看懂这次改动的目的？

如果这两个问题的答案都清楚，这次提交通常就足够好了。
