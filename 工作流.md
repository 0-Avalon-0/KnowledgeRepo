# 知识库工作流

这套方法不依赖 Obsidian。你可以直接使用 VS Code、Typora、任意 Markdown 编辑器，甚至纯文件管理器来维护它。核心思想不是某个软件，而是“原始资料、整理后的知识、派生产出、检查工具”四层分离。

## 你每天怎么用

1. 手动把新资料放进 `原始资料/`。
2. 让 LLM 根据新资料生成或更新 `知识库/` 下的条目。
3. 当你要回答问题、写报告、做总结时，让 LLM 从 `知识库/` 读取并把结果写进 `派生内容/`。
4. 定期检查知识库是否有重复、冲突、缺失链接和过时内容，把结果放进 `检查/`。

## 第一步：手动导入资料

你不需要自动采集，手动导入完全可行。关键是每条资料都尽量保存成“本地、可读、可引用”的文件。

推荐做法：

1. 网页文章放入 `原始资料/文章/`，优先保存成 Markdown，其次是 HTML 或 TXT。
2. 论文 PDF 和摘录放入 `原始资料/论文/`。
3. 代码仓库说明、关键源码片段、仓库摘要放入 `原始资料/仓库/`。
4. 数据表、实验结果、元数据放入 `原始资料/数据集/`。
5. 图表、截图、网页配图放入 `原始资料/图片/`。
6. 尚未成文的摘抄、灵感、聊天记录、临时观察放入 `原始资料/碎片记录/`。
7. 不方便归类的网页存档放入 `原始资料/网页剪藏/`。

导入时建议遵守这几个规则：

1. 文件名尽量可读，例如“2026-04-08-transformer-缩放律.md”。
2. 同一主题的资料尽量放在同一子目录，避免乱堆。
3. 如果一个来源有正文、图片、附件，尽量放在同名或同前缀下，方便 LLM 理解它们属于同一组资料。
4. 碎片记录尽量一条一文件，或者一日一文件，避免把大量不相关内容混进同一个长文档里。

## 碎片知识是否适用

适用，但不能直接把碎片知识当成已经整理好的知识库条目。更稳妥的做法是把碎片知识视作“待编译原料”。

推荐处理方式：

1. 把零散内容先放进 `原始资料/碎片记录/`。
2. 定期让 LLM 对这些碎片做聚类、去重、补充上下文和主题归并。
3. 把整理后的结果分别写入 `知识库/来源笔记/`、`知识库/概念/` 或 `知识库/主题/`。

常见的碎片知识包括：

1. 一句临时灵感。
2. 聊天中的一段高价值解释。
3. 一张截图附带的简单说明。
4. 一段会议记录或语音转写。
5. 一个尚未写成正式文章的观察。

它们的问题通常不是“不能用”，而是缺少上下文、命名混乱、重复率高。所以碎片知识更适合先进入 `原始资料/碎片记录/`，经过整理后再进入 `知识库/`。

示例指令：

```text
请扫描 原始资料/碎片记录/ 下最近新增的内容，按主题聚类，去除重复项，补充必要上下文。
将整理结果写入：
1. 知识库/来源笔记/ 对应的碎片来源笔记
2. 知识库/概念/ 中需要新增或更新的概念页
3. 知识库/索引/待整理清单.md 中仍然无法归类的条目
```

## 第二步：把原始资料编译成知识库

`知识库/` 不是原始摘抄，而是整理后的结构化内容。你可以把这一步理解为“让 LLM 把资料编译成可导航的 Wiki”。

推荐从这三类文件开始：

1. 来源笔记：每个来源一份，放在 `知识库/来源笔记/`。
2. 概念条目：稳定术语和概念，放在 `知识库/概念/`。
3. 主题综述：把多个来源和概念串起来，放在 `知识库/主题/`。

建议的整理顺序：

1. 先让 LLM 为单个来源生成来源笔记。
2. 再让 LLM 从多个来源里抽取重复出现的概念，形成概念条目。
3. 最后让 LLM 按主题编写综述页，并在综述页里链接相关概念和来源。

来源笔记建议至少包含这几类信息：

1. 来源信息：标题、作者、日期、原始资料链接。
2. 内容概览：这个来源主要讲什么。
3. 可靠性与使用方式：它属于什么来源层级，适合支持什么，不适合支持什么。
4. 核心信息或关键事实链：从该来源中可提取的内容。
5. 候选条目与已编入条目：哪些概念和主题只是候选，哪些已经正式沉淀进知识库。
6. 后续整理建议：哪些结论需要补证，哪些概念和主题可以继续编译。

如果你希望固定格式复用，可以直接使用 [工具/来源笔记模板.md](工具/来源笔记模板.md)。新来源导入后，优先基于模板生成来源笔记，再按来源类型删减不必要字段。

来源笔记和概念页、主题页之间不要只依赖语义相似和全文搜索。更稳妥的做法是：

1. 在来源笔记里区分“候选概念 / 候选主题”和“已编入概念 / 已编入主题”。
2. 对已经存在的概念页和主题页，尽量使用显式 Markdown 链接。
3. 只有尚未正式沉淀的条目，才保留为纯文本候选项。

这里要特别区分一件事：来源笔记不是“事实认证书”，而是“对某个来源的结构化整理”。如果原始资料本身可能有误，来源笔记应优先写“该来源声称什么”“该来源适合支持什么”，而不是直接把其中所有说法改写成知识库事实。

更稳妥的准入规则是：

1. 单一二手来源，只先进入来源笔记。
2. 多个相互独立来源重复支持后，再进入概念页或主题页的核心结论。
3. 有争议的判断，在主题页中保留分歧，不写成单一确定结论。

当某个概念页或主题页已经进入 `知识库/` 时，建议同时补一个简短的“证据状态”区块，至少说明三件事：

1. 条目性质，例如“二手概览整理”“经验型交叉整理”“较稳结论”。
2. 当前来源基础，例如由几类来源交叉支持。
3. 使用边界，例如适合做框架整理、术语导航，还是可以支撑更强结论。

这样做的目的，不是把知识页写得更复杂，而是防止 `知识库/` 中的压缩结论被误读成已经充分验证的定论。

你可以直接对代理下这类指令：

```text
请读取 原始资料/文章/ 下新增的文档，为每篇文档在 知识库/来源笔记/ 生成一份来源笔记。
要求：
1. 提取核心观点
2. 记录关键证据和引用片段
3. 标出相关概念
4. 给出应当链接到的主题
```

```text
请基于 知识库/来源笔记/ 和 原始资料/论文/ 的内容，整理一篇“检索增强生成”的主题综述，输出到 知识库/主题/检索增强生成.md。
要求包含：
1. 核心定义
2. 主要方法分类
3. 代表性论文
4. 当前争议点
5. 指向相关概念和来源笔记的链接
```

## 第三步：维护索引，而不是只堆文章

很多知识库失败，不是因为资料太少，而是因为没有入口。`知识库/索引/` 是你和代理都要优先看的地方。

建议至少维护这几类索引：

1. 总索引：列出所有核心主题。
2. 主题地图：按研究方向分组。
3. 来源索引：列出已导入的文章、论文、仓库、数据集。
4. 待整理清单：记录当前真实 backlog，包括未写来源笔记、已写来源笔记但待进一步沉淀，以及已沉淀条目待补高质量来源。

可以让 LLM 定期执行这类任务：

```text
请扫描 原始资料/ 和 知识库/来源笔记/ ，找出尚未整理成来源笔记的原始资料、已完成来源笔记但仍待沉淀的条目，以及已沉淀但仍待补证的知识页，并更新到 知识库/索引/待整理清单.md。
```

索引维护最好作为默认收尾动作，而不是“有空再补”：

1. 每新增主题页时，至少检查一次是否要更新 `知识库/索引/总索引.md` 或对应知识簇索引。
2. 每新增概念页或主题页后，同步检查相关来源笔记中的“已编入条目”和“后续整理建议”是否需要回写。
3. 如果条目的证据基础变化了，也要同步更新对应页中的“证据状态”和待整理清单里的补证项。

## 第四步：提问时不要直接问原始资料，优先问知识库

当你的资料逐渐变多之后，直接让 LLM 每次都读原始资料，效率会越来越差。更稳妥的办法是：

1. 原始资料负责保真。
2. 知识库负责压缩、归纳、连接。
3. 派生内容负责回答当前问题。

也就是说，问答时优先让 LLM 读取 `知识库/`，必要时再回到 `原始资料/` 核对细节。

示例：

```text
请基于 知识库/主题/ 和 知识库/概念/ 回答“为什么长上下文并不能完全替代检索系统”。
如果知识库中证据不足，再回查 原始资料/论文/ 中相关论文。
将结果写入 派生内容/回答/长上下文与检索系统.md。
```

## 第五步：把有价值的问答沉淀回知识库

不是每个回答都值得长期保存，但有些回答会形成新的稳定知识。筛选规则可以很简单：

1. 如果回答只是一次性任务，保留在 `派生内容/回答/`。
2. 如果回答已经形成可复用的主题总结，就整理进 `知识库/主题/`。
3. 如果回答澄清了一个长期会反复出现的术语，就整理进 `知识库/概念/`。
4. 如果回答只是阶段性压缩结果，就放入 `知识库/摘要/`。

这一步很重要，因为它让你的每次查询都在“给知识库增量建设”，而不是查完就丢。

## 第六步：定期做健康检查

当资料增多后，真正影响质量的不是资料量，而是重复、冲突、过期和断链。

建议定期做这些检查，并把结果写入 `检查/`：

1. 重复概念检查：是否有两个概念页其实讲的是同一件事。
2. 孤立条目检查：是否有页面没有被任何索引或主题引用。
3. 来源缺失检查：是否有结论没有回指来源。
4. 过期内容检查：是否有旧结论已被新资料推翻。
5. 待整理积压检查：原始资料是否长期未进入知识库。
6. 状态一致性检查：来源笔记里的“已编入条目”“候选条目”“后续整理建议”是否仍然互相一致。

示例指令：

```text
请检查 知识库/ 下是否存在重复概念、孤立页面、缺少来源支撑的结论，并把问题列表写入 检查/2026-04-08-知识库健康检查.md。
```

## 第七步：推荐的最小人工流程

如果你希望先用最小成本跑起来，直接按下面的节奏执行：

1. 手动把 5 到 20 份资料放进 `原始资料/`。
2. 让 LLM 先只生成来源笔记。
3. 从来源笔记里提炼 10 到 30 个高频概念。
4. 围绕 3 到 5 个主题生成主题综述。
5. 建一个总索引和待整理清单。
6. 开始基于知识库提问，把高价值回答沉淀回去。

这是最稳的起步方式。不要一开始就试图全自动归档几百篇文档，否则目录、命名、索引规则还没稳定，后面会很乱。

## 你在 VS Code 里怎么操作

不使用 Obsidian 时，最简单的组合就是：

1. 用资源管理器浏览目录。
2. 用 Markdown 预览看文档。
3. 用全局搜索查关键词、链接和重复内容。
4. 用代理直接读写这些 Markdown 文件。

你可以把这个仓库当成“给 LLM 和你自己共用的文件系统界面”。软件只是外壳，真正重要的是目录约定、命名规则、索引方式和持续沉淀。

## 一套可直接复制的日常命令思路

每天导入资料后，你可以按这个顺序给代理下任务：

1. “扫描 `原始资料/` 和 `知识库/来源笔记/`，更新 `知识库/索引/待整理清单.md` 中的真实 backlog。”
2. “把待整理清单里还没有来源笔记的资料编写成 `知识库/来源笔记/`。”
3. “从相关来源笔记里提取新概念，合并或更新 `知识库/概念/`，并补写证据状态。”
4. “判断哪些主题页需要更新，并更新 `知识库/主题/`，同时同步索引和来源笔记状态。”
5. “生成今日总结，写入 `知识库/摘要/2026-04-08-今日整理.md`。”
6. “做一次健康检查，输出到 `检查/`。”

## 一个简单判断标准

如果你不确定某个文件该放哪里，就按下面的规则判断：

1. 它是原始材料，没有经过整理：放 `原始资料/`。
2. 它是经过归纳、链接、总结的长期知识：放 `知识库/`。
3. 它是针对某次问题生成的临时结果：放 `派生内容/`。
4. 它是脚本、命令行工具或批处理：放 `工具/`。
5. 它是检查报告或审计结果：放 `检查/`。

## slash 命令怎么用

我已经把这套流程封装成一个 skill，路径在 `.github/skills/knowledge-repo-workflow/SKILL.md`。

使用时可以直接在聊天里输入：

```text
/knowledge-repo-workflow 帮我把 原始资料/文章/ 新增的内容整理进知识库，并同步更新待整理清单与相关索引
```

也可以直接写更明确的任务：

```text
/knowledge-repo-workflow 扫描 原始资料/碎片记录/ ，聚类整理碎片知识，并更新 知识库/概念/ 与 知识库/主题/
```

这个 skill 适合用来执行导入后整理、来源笔记编写、概念提炼、主题更新、问答输出、健康检查，以及碎片知识归并。

## Git 提交与分支规范

这个仓库的本地 Git 使用规则单独整理在 `Git规范.md`。

建议在开始批量导入资料、整理知识库、生成派生报告前，先看一遍这份规范，避免把原始资料、知识条目、派生产物和工具修改混进同一个提交。

## 最后一个关键原则

不要把知识库当成“笔记堆积区”，而要把它当成“持续编译的产物”。

原始资料负责完整保留信息，知识库负责结构化和复用，派生内容负责解决当下问题，检查负责保证质量。只要这四层不混，你不用 Obsidian 也能稳定运行这套方法。
